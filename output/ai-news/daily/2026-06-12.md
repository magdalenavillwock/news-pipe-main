#### Claude Code Releases

**[v2.1.175](https://github.com/anthropics/claude-code/releases/tag/v2.1.175)**
- **`enforceAvailableModels` (neues Managed Setting):** Wenn aktiviert, beschränkt die `availableModels`-Allowlist auch das Default-Modell. Ein Default, der auf ein nicht erlaubtes Modell aufgelöst würde, fällt auf das erste erlaubte Modell zurück. User- und Projekt-Settings können eine verwaltete `availableModels`-Liste damit nicht mehr erweitern. Wichtig für Enterprise-Admins, die strikte Modellkontrolle benötigen.

**[v2.1.174](https://github.com/anthropics/claude-code/releases/tag/v2.1.174)**
- **`wheelScrollAccelerationEnabled`:** Neues Setting zum Deaktivieren der Maus-Scroll-Beschleunigung im Vollbildmodus.
- **Fix `/model`-Picker versteckt Modell-Family:** Opus erscheint jetzt als eigene Zeile auf Max/Team Premium/Enterprise-Plänen; Sonnet auf Pro/Team-Plänen; Opus auf Pay-as-you-go-API-Accounts.
- **Fix `/model`-Picker zeigt falsche Sonnet-Version:** Wenn `ANTHROPIC_DEFAULT_SONNET_MODEL` eine andere Sonnet-Version pinnt, wurde trotzdem die hardgecodete Version angezeigt.
- **Fix Fable-5-Banner falsch angezeigt:** Das Banner „Fable 5 verbraucht jetzt Credits" erschien fälschlicherweise bei Enterprise-Accounts mit nutzungsbasierter Abrechnung.
- **Fix Bedrock GovCloud-Regionen (`us-gov-*`):** Falsches Inference-Profile-Prefix (`global` statt `us-gov`) verursachte 400-Fehler bei abgeleiteten Modell-IDs.
- **Fix Background-Sessions erben fremde Provider-Env-Variablen:** Sessions übernahmen fälschlicherweise `ANTHROPIC_*`-Provider-Umgebungsvariablen (Gateway-URL, Custom Headers, `/model`-Aliase) vom Shell, das den Hintergrund-Daemon gestartet hat.
- **Fix macOS/Linux Exit-Pause:** Nach einem unterbrochenen Shell-Command gab es beim Beenden von Claude Code eine Pause von 1–2 Sekunden.
- **Fix Git-Co-Author-Attribution:** Bei bestimmten Modellen wurde im Commit der falsche Modellname angezeigt.
- **Fix `/advisor`-Dialog:** Vorausgewähltes Modell wurde nicht auf Erlaubtheit gegen `availableModels`-Allowlist geprüft.
- **Fix Skill-Hot-Reload:** Bei einer geänderten Skill-Datei wurde fälschlicherweise die gesamte Skill-Liste neu gesendet; jetzt werden nur geänderte Skills re-announced.
- **Fix Workflow-Tool-Subagents:** Fehlende per-Agent-Attributions-Header in `agent()`-Subagents des Workflow-Tools.
- **[VSCode] Neue Usage-Attribution:** Im Account & Usage Dialog (`/usage`) jetzt Aufschlüsselung nach Cache-Misses, Long Context, Subagents sowie per Skill/Agent/Plugin/MCP für die letzten 24h oder 7 Tage.
- **Fix Pre-warmed Background-Workers:** Scheiterten mit „Could not resolve authentication method", wenn sie nach Idle-Zeit beansprucht wurden.

#### AWS AI Blog

**[Agent-EvalKit: Systematische KI-Agenten-Evaluation](https://aws.amazon.com/blogs/machine-learning/evaluate-ai-agents-systematically-with-agent-evalkit/)**
AWS veröffentlicht Agent-EvalKit (Apache 2.0), ein Open-Source-Toolkit zur strukturierten Evaluation von KI-Agenten direkt im Entwicklungsworkflow. Es integriert sich mit Claude Code, Kiro CLI und Kilo Code. Das Toolkit arbeitet in 6 Phasen:
1. **Plan** (`/evalkit.plan`): Analysiert Agent-Code und entwirft Evaluierungsplan
2. **Data** (`/evalkit.data`): Generiert Testfälle mit Ground-Truth-Outcomes
3. **Trace** (`/evalkit.trace`): Fügt OpenTelemetry-kompatibles Tracing hinzu
4. **Run Agent** (`/evalkit.run_agent`): Führt Testfälle aus, sammelt Traces
5. **Eval** (`/evalkit.eval`): Implementiert Metriken und bewertet Traces
6. **Report** (`/evalkit.report`): Priorisierte Code-spezifische Verbesserungsempfehlungen

Fallstudie: Ein Reise-Recherche-Agent zeigte 83,9% Response Quality aber nur 32,3% Faithfulness — der Agent halluzinierte Wechselkurse und Temperaturen bei leeren Tool-Ergebnissen.

#### VentureBeat

**[Microsoft SkillOpt: KI-Agent-Skills automatisch optimieren](https://venturebeat.com/orchestration/microsofts-open-source-skillopt-automatically-upgrades-ai-agent-skills-without-touching-model-weights)**
SkillOpt (MIT-Lizenz, Open Source) wendet Deep-Learning-Konzepte auf die Optimierung von Agent-Skill-Dokumenten (.md-Dateien) an — ohne die Modellgewichte zu ändern. Das System:
- Nutzt ein iteratives Propose-and-Test-Loop mit separatem Optimizer-Modell
- Implementiert "Learning Rate" via Edit-Budget (max. Änderungen pro Schritt)
- Nutzt Held-Out Validation Gates, um Regressionen zu verhindern
- Speichert abgelehnte Edits in einem Negativ-Buffer (kein Wiederholen von Fehlern)

Ergebnisse: +23,5 Punkte durchschnittlich über No-Skill-Baseline bei GPT-5.5; GPT-5.4-nano verdoppelte seinen Score bei multimodaler Dokument-QA. Trainingskost für eine einzelne Task: ~$1–5.

**[Xiaomi MiMo Code: Claude Code auf langen Tasks geschlagen](https://venturebeat.com/technology/xiaomis-new-open-source-agentic-ai-coding-harness-mimo-code-beats-claude-code-at-ultra-long-200-step-tasks)**
MiMo Code V0.1.0 (Open Source, MIT-Lizenz, Fork von OpenCode) mit Fokus auf Cross-Session-Memory:
- SQLite FTS5-basiertes Gedächtnissystem (4 Ebenen: Projekt-Memory, Session-Checkpoints, Scratch-Notes, Task-Progress)
- Unabhängiger Checkpoint-Writer-Subagent dokumentiert parallel
- `/dream`-Command für periodische Memory-Konsolidierung
- Installierbar via `curl -fsSL https://mimo.xiaomi.com/install | bash`

**Selbstgemeldete Benchmarks** (nicht unabhängig verifiziert):
- SWE-bench Verified: 82% vs. Claude Code 79%
- SWE-bench Pro: 62% vs. 55%
- Terminal Bench 2: 73% vs. 69%
- Bei 200+ Schritten: Gewinnrate >65% vs. Claude Code (interner A/B-Test, 576 Entwickler)

Preis: MiMo-V2.5 ab $0,40/$2,00 pro Mio. Tokens (Input/Output). Aktuell: kostenloser Zugang zu MiMo-V2.5 ohne Registrierung.

**[Google DiffusionGemma: Parallele Textgenerierung](https://venturebeat.com/technology/googles-diffusiongemma-generates-256-tokens-in-parallel-and-self-corrects-as-it-goes)**
26B MoE-Modell (3,8B aktive Parameter), Apache 2.0. Generiert 256-Token-Blöcke parallel statt sequenziell:
- H100 FP8: 1.008 Token/Sekunde (~5× autoregressive Baseline)
- H200: 1.288 Token/Sekunde (~6×)
- Selbstkorrektur: Unsichere Token-Positionen werden in nächstem Pass neu bewertet
- Bidirektionaler Kontext: Jede Position sieht alle anderen gleichzeitig
- Einschränkung: Google räumt niedrigere Output-Qualität gegenüber Standard-Gemma 4 ein
- Ideal für: Lokale Inferenz, low-concurrency Deployments, Code-Infilling, strukturierte Generierung

**[LCLM: 16× Kontext-Kompression ohne Genauigkeitsverlust](https://venturebeat.com/data/context-compression-finally-works-in-production-new-research-cuts-llm-input-16x-without-the-accuracy-hit)**
Forschung von NYU/Columbia/Princeton/Maryland/Harvard/LLNL: Latent Context Language Models komprimieren Input-Tokens vor dem Decoder-Prefill (nicht KV-Cache-Eviction danach).
- 4× Kompression: 91,76% Genauigkeit (vs. 94,41% unkomprimiert)
- 16× Kompression: 75,06% Genauigkeit — 8,8× schneller als KV-Cache-Baseline
- Architektur: 0,6B Encoder + 4B Decoder, 350 Mrd. Token Training
- Code: github.com/LeonLixyz/LCLM, Modelle: huggingface.co/latent-context

**[EurekAgent: Agent Environment Engineering für autonome Wissenschaft](http://arxiv.org/abs/2606.13662v1)** — Argumen, dass der Bottleneck für autonome wissenschaftliche Entdeckung sich von Agenten-Workflows hin zu Agenten-Umgebungsdesign verschiebt.

**[SpatialClaw: Action Interface für räumliches Reasoning](http://arxiv.org/abs/2606.13673v1)** — Untersucht, wie das Design der Action-Interface (wie Agenten Tool-Calls formulieren) die räumliche Reasoning-Fähigkeit von VLMs grundlegend beeinflusst.
